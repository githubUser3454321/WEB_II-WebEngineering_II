Ausgangslage

Im bisherigen Projekt wurden GUI (Frontend), RESTful Web Service (Backend) sowie eine relationale Datenbank (Persistenz) implementiert (Sprint 01–04).

Die Geschäftsleitung möchte nun prüfen, ob die Applikation produktionsnah betrieben werden kann. Dazu soll die Architektur realitätsnah verteilt werden.

Eine monolithische Struktur (alles lokal) genügt nicht mehr den Anforderungen an Skalierbarkeit, Sicherheit und Wartbarkeit.

Die Applikation soll neu aus drei getrennten Schichten bestehen:

Frontend (GUI: Presentation Tier)

Backend (RESTful Web Service / Business-Service / API-Schicht: Business Tier)

Persistence (Datenbank: EIS Tier)

Diese drei Teile müssen auf unterschiedlichen Plattformen betrieben werden.

Verfügbare Plattformen

Microsoft Azure

Ubuntu Linux VM (VMware, vmnet1)

Zugriff via pfSense Firewall

Ausgehend ins Internet: erlaubt

Portweiterleitungen über pfSense: erlaubt

Lokale Umgebung (Laptop)

Die Plattformwahl ist frei.

Wichtig: Jede Schicht muss auf einer anderen Plattform laufen.

Beispiele:

Frontend: lokal – Backend: Azure – Persistence: Ubuntu

Frontend: Azure – Backend: Ubuntu – Persistence: lokal

Arbeitsauftrag 2 (Partnerarbeit)

Praxis – Umsetzung der verteilten Systemarchitektur.

Im ersten Arbeitsauftrag haben Sie ein technisches Konzept für den verteilten Betrieb Ihrer Applikation erstellt und begründet. Nun wird dieses Konzept produktiv umgesetzt.

Ziel ist es, die drei definierten Schichten:

Frontend (Presentation Tier)

Backend (Business Tier)

Persistence (EIS Tier)

auf drei unterschiedlichen Plattformen zu betreiben und technisch korrekt miteinander zu verbinden.

Die definierten Sicherheits- und Netzwerkvorgaben müssen eingehalten werden.

Die Umsetzung muss so dokumentiert sein, dass eine Drittperson die komplette Umgebung ohne zusätzliche Recherche nachbauen kann.

Aufgabe 2.1 – Vorbereitung der Plattformen

Richten Sie die drei gewählten Plattformen gemäss Ihrem Konzept ein.

Umzusetzende Punkte:

Installation der benötigten Laufzeitumgebungen
(z. B. Node.js, NPM, Python, DBMS usw.)

Einrichten der Netzwerkverbindungen

Konfiguration der Firewall-Regeln (insbesondere bei pfSense)

Sicherstellen, dass die Datenbank nicht öffentlich erreichbar ist

Falls Portweiterleitungen notwendig sind, müssen diese sauber dokumentiert werden.

Erwartetes Resultat

drei lauffähige Umgebungen gemäss Architekturkonzept

Netzwerkverbindungen funktionieren

Datenbank ist nur intern erreichbar

dokumentierte Firewall- und Portkonfiguration

Aufgabe 2.2 – Deployment der Datenbank

Installieren und konfigurieren Sie die Datenbank auf der gewählten Plattform.

Die Datenbank muss:

mindestens drei Tabellen enthalten

mindestens eine Beziehung (Foreign Key) enthalten

Testdaten enthalten

nur vom Backend erreichbar sein

Überprüfen Sie die Erreichbarkeit gezielt mit und ohne Berechtigung.

Erwartetes Resultat

funktionsfähige relationale Datenbank

ER-Diagramm oder tabellarische Struktur

Nachweis, dass kein externer Zugriff möglich ist

dokumentierte Installations- und Konfigurationsschritte

Aufgabe 2.3 – Deployment der Business-Logik

Deployen Sie die Business- / Service-Schicht auf der vorgesehenen Plattform.

Diese Schicht muss:

mit der Datenbank kommunizieren

mindestens eine fachliche Regel implementieren
(z. B. Validierung, Berechnung, Statuswechsel)

Fehler sauber behandeln

Fehler protokollieren

Verwenden Sie keine im Code hinterlegten Zugangsdaten.

Secrets müssen über:

Umgebungsvariablen
oder

eine gesicherte Konfigurationsdatei

eingebunden werden.

Erwartetes Resultat

Backend verbindet sich erfolgreich mit der DB

Business-Regel funktioniert nachweislich

keine sensiblen Daten im Code

Fehler werden geloggt
